§ 01

The Launcher Stack

A launcher is a privileged system application — it holds BIND_APPWIDGET permission, receives ACTION_MAIN + CATEGORY_HOME intent and sits one layer above the WindowManager. Understanding the full stack is essential before customizing behavior.

Key Insight
Launcher3 introduced LauncherModel as a background thread that owns all data loading — it uses a LoaderTask (Runnable) to load workspace, AllApps and widget data off the main thread, then dispatches results via callbacks. Never touch model data on the UI thread.
§ 02

Interacting with Installed Apps

Every icon on your launcher maps to a WorkspaceItemInfo (Launcher3) or ShortcutInfo (Launcher2). The data pipeline from PackageManager to icon grid is more involved than it looks.

The Intent that makes you a launcher

A launcher registers itself in the manifest with a specific intent filter. This is what Android uses to recognize home screen candidates and show the "Set as default" dialog.

    public int runtimeStatusFlags;
}

App loading pipeline

Launcher3's LoaderTask loads in two phases to give the user a fast first paint:

LoaderTask.java — simplified flow 
// Phase 1: load workspace from DB — fast, runs first
private void loadWorkspace() {
    // Read from launcher.db (favorites table)
    Cursor c = queryWorkspace();
    while (c.moveToNext()) {
        int itemType = c.getInt(iconTypeIndex);
        switch (itemType) {
            case Favorites.ITEM_TYPE_APPLICATION:
                info = getAppShortcutInfo(intent, user, pmHelper);
                break;
            case Favorites.ITEM_TYPE_APPWIDGET:
                info = getWidgetInfo(appWidgetId);
                break;
            case Favorites.ITEM_TYPE_FOLDER:
                info = new FolderInfo();
                break;
        }
    }
}

// Phase 2: load AllApps list — all installed apps
private void loadAllApps() {
    List<LauncherActivityInfo> apps =
        mLauncherApps.getActivityList(null, user);

    // IconCache fills in labels + icons async, batched by package
    for (LauncherActivityInfo app : apps) {
        AppInfo appInfo = new AppInfo(app, user, isQuietModeEnabled);
        mIconCache.getTitleAndIcon(appInfo, app, false);
        mBgAllAppsList.add(appInfo);
    }
}

Responding to package events in real time

PackageUpdatedTask.java 
// Launcher3 handles package changes via a dedicated task
// This runs on the model thread (not main thread)
public class PackageUpdatedTask extends BaseModelUpdateTask {

    private final int mOp; // OP_ADD, OP_REMOVE, OP_UPDATE, OP_SUSPEND

    @Override
    public void execute(LauncherAppState app, BgDataModel dataModel, ...) {
        switch (mOp) {
            case OP_ADD:
                // Add to AllApps, predict if ML model suggests it
                dataModel.allAppsList.addPackage(context, packageName, user);
                break;
            case OP_REMOVE:
                // Remove icons, clean workspace, remove from DB
                deleteAndBindComponentsRemoved(removedComponents);
                break;
            case OP_UPDATE:
                // Re-fetch icon, label — clear IconCache entry first
                mIconCache.invalidateCacheEntries(packageName, user);
                updateWorkspaceItems(updatedItems);
                break;
        }
    }
}
Performance Note
IconCache is one of the most critical components for a smooth launcher. It uses a two-level cache: an in-memory LRU (BitmapInfo objects) and a persistent SQLite-backed icon DB (app_icons.db). Always invalidate stale entries before re-querying; a stale icon DB is a common source of launch-time lag.
§ 03

Widget System Internals

Widgets are the most complex part of any launcher. They run in a remote process, communicate via RemoteViews and require the launcher to be a trusted AppWidgetHost.

AppWidgetHost
Lives in your launcher process. Creates AppWidgetHostView containers and owns the communication channel with AppWidgetService via Binder.
AppWidgetProvider
BroadcastReceiver in the widget's app. Sends RemoteViews to AppWidgetService, which relays to your Host.
AppWidgetService
System service (IPC boundary). Mediates between Host and Provider, manages widget IDs and handles bind/unbind lifecycle.
RemoteViews
Parcelable view tree. Can only use a subset of Views (TextView, ImageView, etc.) — no custom views, no direct touch listeners.
LauncherAppWidgetHost.java 
// Always use a fixed HOST_ID — widgets are bound to it in DB
public static final int APPWIDGET_HOST_ID = 1024;

public class LauncherAppWidgetHost extends AppWidgetHost {

    public LauncherAppWidgetHost(Context context) {
        super(context, APPWIDGET_HOST_ID);
    }

    @Override
    protected AppWidgetHostView onCreateView(Context context,
            int appWidgetId, AppWidgetProviderInfo appWidget) {
        // Return custom view — add drag handles, resize frames, etc.
        return new LauncherAppWidgetHostView(context);
    }

    @Override
    public void onProvidersChanged() {
        // Widget providers installed/removed — refresh picker
        mLauncher.notifyWidgetProvidersChanged();
    }
}

// Lifecycle — CRITICAL: start/stop must be symmetric
void onResume()  { mAppWidgetHost.startListening(); }
void onPause()   { mAppWidgetHost.stopListening(); }
void onDestroy() {
    // Only call deleteHost() if you want to remove ALL bound widgets
    // Usually you DON'T — just stopListening()
}

Binding a widget — the full flow

WidgetAddFlowHandler.java 
// Step 1: allocate a widget ID
int appWidgetId = mAppWidgetHost.allocateAppWidgetId();

// Step 2: attempt to bind (requires BIND_APPWIDGET permission)
AppWidgetManager awm = AppWidgetManager.getInstance(context);
boolean bindAllowed = awm.bindAppWidgetIdIfAllowed(
    appWidgetId, providerInfo.provider
);

if (!bindAllowed) {
    // User hasn't granted permission — launch system dialog
    Intent intent = new Intent(AppWidgetManager.ACTION_APPWIDGET_BIND)
        .putExtra(AppWidgetManager.EXTRA_APPWIDGET_ID, appWidgetId)
        .putExtra(AppWidgetManager.EXTRA_APPWIDGET_PROVIDER, providerInfo.provider);
    startActivityForResult(intent, REQUEST_BIND_APPWIDGET);
}

// Step 3: check if widget needs configuration Activity
if (providerInfo.configure != null) {
    Intent configIntent = new Intent(AppWidgetManager.ACTION_APPWIDGET_CONFIGURE)
        .setComponent(providerInfo.configure)
        .putExtra(AppWidgetManager.EXTRA_APPWIDGET_ID, appWidgetId);
    startActivityForResult(configIntent, REQUEST_CREATE_APPWIDGET);
}

// Step 4: inflate and add to workspace
AppWidgetHostView hostView = mAppWidgetHost.createView(context, appWidgetId, providerInfo);
mWorkspace.addInScreen(hostView, itemInfo);
Common Pitfall
Always call deleteAppWidgetId() on IDs you allocated but couldn't bind — orphaned widget IDs accumulate in the system's widget registry and are never garbage collected. On devices with many launcher installs/uninstalls this becomes a real problem.
§ 04

Workspace, DragLayer & CellLayout

The workspace is a PagedView (horizontal scroll) containing multiple CellLayout pages. Every item on screen is placed into a cell grid — the grid math is non-trivial once you account for folders, spans and RTL.

Component Role Key methods
DragLayer Full-screen overlay. Intercepts all touch events during drag. Hosts drag preview ("drag view"). performDrag · animateViewIntoPosition · getDescendantRectRelativeToSelf
Workspace PagedView subclass. Manages pages, handles drop targets, owns item-to-page mapping. addInScreen · removeItemFromScreen · snapToPage · getPageAt
CellLayout One page of the workspace grid. Manages cell occupancy, reorder animations. findCellForSpan · addViewToCellLayout · animateChildToPosition · markCellsAsOccupied
ShortcutAndWidgetContainer Direct parent of all icons + widgets within a CellLayout. Handles measure/layout with cell math. getChildAt(cellX, cellY) · measureChild · layoutDecorated
Hotseat Bottom persistent row. Single CellLayout with rank-based ordering. getLayout · mHotseatGrid · isHotseatLayout

Cell position math

CellLayout.java — cell coordinate helpers 
// Convert pixel coordinates → cell coordinates
public void pointToCellExact(int x, int y, int[] result) {
    final int hStartPadding = getPaddingLeft();
    final int vStartPadding = getPaddingTop();
    result[0] = (x - hStartPadding) / (mCellWidth  + mWidthGap);
    result[1] = (y - vStartPadding) / (mCellHeight + mHeightGap);
    // Clamp to grid bounds
    result[0] = Math.max(0, Math.min(result[0], mCountX - 1));
    result[1] = Math.max(0, Math.min(result[1], mCountY - 1));
}

// Find a free cell for an item of given span
public boolean findCellForSpan(int[] cellXY, int spanX, int spanY) {
    return findCellForSpanThatIntersects(cellXY, spanX, spanY, -1, -1);
}

// Check if a region is occupied
public boolean isRegionVacant(int x, int y, int spanX, int spanY) {
    for (int i = x; i < x + spanX; i++) {
        for (int j = y; j < y + spanY; j++) {
            if (mOccupied[i][j]) return false;
        }
    }
    return true;
}

Drag & Drop lifecycle

DragController.java — key events 
// Start a drag — called on long-press via ItemLongClickListener
public DragView startDrag(View v, DragSource source, ...) {
    // 1. Create bitmap snapshot of the view
    Bitmap dragBitmap = BitmapRenderer.createHardwareBitmap(v);
    // 2. Inflate DragView (the floating preview)
    mDragView = new DragView(mLauncher, dragBitmap, registrationX, registrationY);
    mDragView.show(initialX, initialY);
    // 3. Hide original, dispatch onDragStart to all listeners
    v.setVisibility(View.INVISIBLE);
    dispatchDragStart(mDragging, dragAction);
    return mDragView;
}

// DragListener implemented by Workspace, Hotseat, DeleteDropTarget, etc.
interface DropTarget {
    void onDragEnter(DragObject d);
    void onDragOver(DragObject d);   // real-time hover feedback
    void onDragExit(DragObject d);
    boolean acceptDrop(DragObject d); // can this target accept the item?
    void onDrop(DragObject d, DragOptions options);
}
Architecture Note
In Launcher3, the DragLayer is the single coordinator for all drag operations — not the individual pages. This is why touch events during drag must be funneled through it even if the visual target is in a different CellLayout. When building custom drop targets (e.g. a custom folder type or app drawer zone), implement DropTarget and register with DragController.addDropTarget().
